## Методологія

### User Stories

Користувацькі історії створені за форматом:
```
As a [роль]
I want [функціонал]
So that [цінність/вигода]
```

### Acceptance Criteria (Gherkin)

Критерії приймального тестування написані мовою Gherkin:
```gherkin
Feature: [назва функції]

  Scenario: [конкретний сценарій]
    Given [передумова]
    And [додаткова передумова]
    When [дія користувача]
    Then [очікуваний результат]
    And [додатковий результат]
```

**Ключові слова Gherkin:**
- **Feature** - опис функції/можливості
- **Scenario** - конкретний сценарій використання
- **Given** - передумови (початковий стан)
- **When** - дія користувача або подія
- **Then** - очікуваний результат
- **And** - додаткова умова
- **But** - виняток

### Traceability Matrix

Матриця відстежуваності забезпечує **повну простежуваність** від бізнес-вимог до критеріїв тестування:

**Use Case** (Lab 2) → **FR** (Lab 3) → **NFR** (Lab 3) → **Epic** (Lab 4) → **User Story** (Lab 4) → **Acceptance Criteria** (Lab 4) → **Prototype** (Lab 4)

### Горизонтальні прототипи

**Горизонтальний прототип** - це UI wireframe, який показує зовнішній вигляд інтерфейсу без реалізації backend логіки.

**Характеристики:**
- Візуалізація основних екранів
- Розташування елементів інтерфейсу
- Навігація між екранами
- Не містить робочого коду
- Використовуються для валідації вимог з замовником

**Відмінність від вертикальних прототипів:**
- **Горизонтальний** - широкий функціонал, мінімальна глибина (багато екранів, без логіки)
- **Вертикальний** - вузький функціонал, максимальна глибина (один екран, повна логіка)

## Контрольні запитання

### 1. Визначте дії, які необхідно виконати для забезпечення перевірки вимог відповідно з областю знань "Перевірка вимог" в SWEBOK?

Відповідно до SWEBOK (Software Engineering Body of Knowledge), перевірка вимог включає наступні дії:

**Inspection (Інспектування):**
- Формальний огляд документів вимог
- Перевірка на дотримання стандартів та шаблонів
- Виявлення дефектів та неузгодженостей

**Reviews (Огляди):**
- Peer review - огляд колегами
- Walkthrough - презентація вимог командою
- Technical review - технічний огляд експертами

**Prototyping (Прототипування):**
- Створення горизонтальних прототипів (UI/UX)
- Створення вертикальних прототипів (функціонал)
- Отримання зворотного зв'язку від стейкхолдерів

**Model Validation (Валідація моделей):**
- Перевірка моделей вимог (діаграми, специфікації)
- Аналіз узгодженості між різними моделями
- Валідація повноти та коректності

**Acceptance Tests (Приймальне тестування):**
- Розробка критеріїв приймання
- Створення тестових сценаріїв
- Валідація з користувачами

**Traceability Analysis (Аналіз простежуваності):**
- Побудова Traceability Matrix
- Перевірка покриття всіх вимог
- Виявлення orphan requirements (вимоги без зв'язків)

### 2. Визначте відмінності між термінами перевірка (verification) і утвердження (validation) вимог?

**Verification (Перевірка)** - "Are we building the product right?"
- Перевірка чи продукт відповідає специфікації
- Відповідність стандартам та регуляціям
- Технічна коректність вимог
- Методи: інспектування, огляди, статичний аналіз
- Питання: "Чи правильно ми реалізували вимоги?"

**Validation (Утвердження)** - "Are we building the right product?"
- Перевірка чи продукт задовольняє потреби користувача
- Відповідність бізнес-цілям
- Користувацька цінність
- Методи: прототипування, демо, приймальне тестування
- Питання: "Чи те, що ми будуємо, потрібно користувачеві?"

**Приклад:**
- **Verification**: Система дозволяє авторизацію через PIN-код з 6 цифр (відповідає специфікації FR1.2)
- **Validation**: Користувачі успішно входять в систему і вважають процес зручним (задовольняє потреби)

**У цій лабораторній роботі:**
- **Verification**: Traceability Matrix (перевірка покриття всіх FR)
- **Validation**: Горизонтальні прототипи (валідація UI з користувачами), Acceptance Criteria (критерії приймання)

### 3. Надайте перелік ключових слів мови Gherkin?

**Основні ключові слова Gherkin:**

1. **Feature** - Опис функції/можливості системи
2. **Scenario** - Конкретний сценарій використання
3. **Scenario Outline** - Шаблон сценарію з параметрами
4. **Examples** - Таблиця з даними для Scenario Outline
5. **Given** - Передумови (початковий стан системи)
6. **When** - Дія користувача або подія
7. **Then** - Очікуваний результат
8. **And** - Додаткова умова (для Given/When/Then)
9. **But** - Виняток або контрастна умова
10. **Background** - Загальні передумови для всіх сценаріїв у Feature
11. **Rule** - Бізнес-правило (Gherkin 6+)
12. **Example** - Альтернатива для Scenario

**Додаткові ключові слова:**
- `#` - коментар
- `"""` - багаторядковий текст (Doc Strings)
- `|` - таблиця з даними (Data Tables)
- `@tag` - тег для групування сценаріїв

**Приклад використання:**
```gherkin
Feature: Реєстрація працівника

  Background:
    Given система SafeHeight Monitor активна

  @critical @authentication
  Scenario: Успішна реєстрація через QR-код
    Given працівник має виданий IoT пристрій
    And працівник має дійсний обліковий запис
    When працівник сканує QR-код
    Then система ідентифікує працівника
    And статус змінюється на 'Активний'
    But час реєстрації не перевищує 15 секунд
```

### 4. Назвіть методи обговорення документу з вимогами?

**Формальні методи:**

1. **Inspection (Інспектування Фагана)**
   - Формальний процес з визначеними ролями (moderator, reader, recorder)
   - Попередня підготовка учасників
   - Виявлення дефектів з їх класифікацією
   - Найефективніший, але найдорожчий метод

2. **Formal Technical Review**
   - Офіційна зустріч з технічними експертами
   - Детальна перевірка документації
   - Протокол зустрічі з рішеннями

3. **Audit (Аудит)**
   - Незалежна оцінка документів третьою стороною
   - Перевірка відповідності стандартам

**Неформальні методи:**

4. **Walkthrough (Прохід)**
   - Автор презентує вимоги команді
   - Неформальне обговорення
   - Збір зауважень та пропозицій

5. **Peer Review (Огляд колегами)**
   - Неформальна перевірка колегою
   - Швидкий фідбек
   - Низька вартість

6. **Pair Programming Review**
   - Два розробники працюють разом
   - Обговорення вимог в процесі роботи

**Сучасні Agile методи:**

7. **Sprint Review / Demo**
   - Демонстрація функціоналу стейкхолдерам
   - Збір зворотного зв'язку
   - Адаптація вимог

8. **Three Amigos Meeting**
   - Зустріч Product Owner + Developer + QA
   - Обговорення User Story та Acceptance Criteria
   - Уточнення деталей перед розробкою

9. **Backlog Refinement (Grooming)**
   - Регулярний перегляд та деталізація вимог
   - Оцінка складності
   - Пріоритизація

**Інші методи:**

10. **Prototyping Sessions**
    - Обговорення вимог через прототипи
    - Візуалізація ідей
    - Валідація з користувачами

11. **Stakeholder Interviews**
    - Індивідуальні інтерв'ю зі стейкхолдерами
    - Глибоке розуміння потреб

12. **Workshops / JAD Sessions (Joint Application Development)**
    - Групові робочі сесії
    - Швидке досягнення консенсусу
    - Вирішення конфліктів вимог

**У цьому проекті використані:**
- Traceability Matrix для formal review
- Горизонтальні прототипи для prototyping sessions
- Gherkin scenarios для Three Amigos approach
