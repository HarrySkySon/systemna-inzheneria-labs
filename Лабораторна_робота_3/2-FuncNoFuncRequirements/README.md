## Проект

**Назва:** SafeHeight Monitor - Веб-додаток моніторингу будівельних майданчиків з AI для запобігання падінням з висоти

**Опис:** Система використовує IoT пристрої та штучний інтелект для моніторингу стану працівників будівельних майданчиків в реальному часі з метою виявлення та попередження ризиків падіння з висоти.

## Структура лабораторної роботи

### [2.1 Функціональні вимоги](./2.1-FunctionalRequirements/)

Документування функціональних вимог (FR) до системи SafeHeight Monitor відповідно до стандарту ISO/IEC/IEEE 29148:2011.

**Загальна кількість вимог:** 12 груп функціональних вимог (FR1-FR12) з 58 деталізованими підвимогами

**Основні категорії:**
- FR1: Реєстрація та автентифікація користувачів
- FR2: Моніторинг працівників в реальному часі
- FR3: Виявлення та попередження про ризики падіння з висоти
- FR4: Керування робочими зонами
- FR5: Налаштування правил безпеки
- FR6: Керування IoT пристроями
- FR7: Керування користувачами
- FR8: Звітність та аналітика
- FR9: Фільтрація та пошук
- FR10: Підтримка офлайн режиму
- FR11: Аварійний протокол
- FR12: Логування подій

### [2.2 Нефункціональні вимоги](./2.2-NonFunctionalRequirements/)

Документування нефункціональних вимог (NFR) з прив'язкою до функціональних вимог (FR) відповідно до стандарту ISO/IEC/IEEE 29148:2011.

**Загальна кількість вимог:** 65 нефункціональних вимог (NFR)

**Основні категорії:**
- Продуктивність (Performance)
- Надійність (Reliability)
- Безпека (Security)
- Юзабіліті (Usability)
- Масштабованість (Scalability)
- Сумісність (Compatibility)
- Доступність (Availability)
- Ємність (Capacity)
- Переносимість (Portability)
- Підтримка та обслуговування (Maintainability)
- Стійкість до збоїв (Fault Tolerance)
- Енергоефективність (Energy Efficiency)
- Юридичні та етичні вимоги (Legal and Ethical)
- Організаційні вимоги (Organizational)

## Методологія

Вимоги розроблені на основі:
1. **Use Cases** з Лабораторної роботи №2 (11 варіантів використання)
2. **Аналізу бізнес-цілей** з Лабораторної роботи №1
3. **Стандарту ISO/IEC/IEEE 29148:2011** - Systems and software engineering - Requirements engineering

### Критерії якості функціональних вимог

Всі функціональні вимоги відповідають критеріям:
1. **Атомарність** (Singular) - одна вимога описує одну функцію
2. **Завершеність** (Complete) - вимоги описані повністю без необхідності додаткової деталізації
3. **Необхідність** (Necessary) - кожна вимога є необхідною для функціонування системи
4. **Однозначність** (Unambiguous) - термінологія зрозуміла всім стейкхолдерам
5. **Відокремленість від реалізації** (Implementation Free) - описано ЩО, а не ЯК
6. **Узгодженість** (Consistent) - відсутність протиріч між вимогами
7. **Здійсненність** (Feasible) - технічна можливість реалізації
8. **Перевіряємість** (Verifiable) - наявність критеріїв перевірки
9. **Відстежуваність** (Traceable) - унікальні ієрархічні ідентифікатори

## Контрольні запитання

1. **Які документи відповідно до SWEBOK включені в список рекомендованих для опису комплексних проектів з розробки програмного забезпечення?**

   Відповідно до SWEBOK (Software Engineering Body of Knowledge), для опису комплексних проектів рекомендовані такі документи:
   - Software Requirements Specification (SRS)
   - Software Design Description (SDD)
   - Software Test Documentation
   - Software User Documentation
   - Software Configuration Management Plan

2. **Який стандарт описує структуру документа «специфікація програмних вимог»?**

   Структура документа Software Requirements Specification (SRS) описана в стандарті **IEEE 830-1998** (пізніше замінений на **ISO/IEC/IEEE 29148:2011**).

3. **Які дії необхідно виконувати для «управління змінами» до вимог?**

   - Реєстрація запиту на зміну вимоги
   - Аналіз впливу зміни на систему
   - Оцінка вартості та термінів
   - Затвердження або відхилення зміни
   - Оновлення документації вимог
   - Відстеження версій вимог
   - Повідомлення стейкхолдерів про зміни

4. **Що означає базова версія вимог?**

   Базова версія (baseline) вимог - це офіційно затверджена версія документа вимог, яка служить основою для подальшої розробки. Будь-які зміни до базової версії проходять через формальний процес управління змінами.

5. **Які варіанти документування змін до вимог ви знаєте?**

   - Система контролю версій (Git, SVN)
   - Журнал змін (Change Log)
   - Матриця відстеження змін (Traceability Matrix)
   - Документ з історією ревізій
   - Спеціалізовані інструменти управління вимогами (JIRA, Azure DevOps, IBM DOORS)
